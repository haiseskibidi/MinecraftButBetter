2. **ОБЯЗАТЕЛЬНОЕ ПЕРЕЧИТЫВАНИЕ:** Ты ОБЯЗАН перечитывать файлы `architecture.md`, `code_map.md` и `current_state.md` ПЕРЕД ТЕМ, как начинать любую задачу. Это необходимо для актуализации знаний о структуре классов и текущем прогрессе без лишнего поиска по коду.
3. **MCP Knowledge Graph:** Для хранения сложных связей между системами используется MCP `memory`. Ты ОБЯЗАН синхронизировать архитектурные изменения с графом знаний.
3. **Протокол:** Любое изменение в коде, лоре или планах ОБЯЗАНО быть синхронизировано с соответствующим файлом памяти через навык `update_memory`.
4. **Хирургические правки:** При обновлении файлов (как кода, так и памяти) используй точечные изменения. ЗАПРЕЩЕНО полностью перезаписывать файл, если нужно добавить или изменить только часть информации.
5. Ты не полагаешься на историю чата — ты полагаешься на файлы и граф знаний.
6. Раздумывание, общение и тп на русском языке!!!

# Стандарты разработки (Lead Developer Mandates)

### 1. Работа с индексами ресурсов (КРИТИЧЕСКИ ВАЖНО)
- **ЗАПРЕЩЕНО** полностью перезаписывать `.index` файлы. Это **Append-only** ресурсы.
- **Журнал (Journal)**: Для появления статьи в игре необходимо:
    1. Создать JSON в `journal/entries/`.
    2. Добавить имя файла в `journal/entries/.index`.
    3. Добавить Identifier статьи в массив `entries` соответствующей категории в `journal/categories/`.
    4. Проверить наличие ключей в `lang/ru_ru.json`.

### 2. Принцип Data-Driven Development
- **ЗАПРЕЩЕНО** хардкодить игровые данные (параметры блоков, статы предметов, рецепты, планировку зданий) в Java-коде.
- Все данные должны храниться в JSON-файлах в папке `src/main/resources/<namespace>/`.
- Для загрузки использовать `DataLoader`. Новые типы данных должны регистрироваться в соответствующих реестрах (`Registry`, `NumericalRegistry`).

### 2. Система Идентификации и Реестров
- **Identifier**: Всегда используй `Identifier.of("namespace:path")` для обращения к ресурсам. Никаких магических чисел.
- **Классы-холдеры**: Для удобного доступа к "ванильному" контенту используй классы `Blocks`, `Items` и `PrefabManager`. 
    - Чтобы добавить новый блок/предмет для быстрого доступа, просто объяви `public static` поле. Оно заполнится автоматически через рефлексию при запуске.
- **Динамические ID**: Никогда не полагайся на фиксированные числовые ID (int/byte). Они назначаются реестром динамически.

### 3. Компонентная архитектура (Data Components)
- Избегай глубокого наследования классов. 
- Расширяй функционал предметов через `ItemComponent` (`FoodComponent`, `ToolComponent` и т.д.).
- Логика взаимодействия должна проверять наличие компонента, а не класс предмета.

### 4. Расширяемость Логики
- **PlacementType**: При создании новых типов блоков (плиты, лестницы) используй `PlacementType` в JSON. Не хардкодь логику установки в `InputManager`.
- **Registry Events**: Используй `RegistryEvents` для подключения модов к фазам регистрации контента.

### 5. LLM-friendliness и Качество кода
- Пиши подробные JSDoc/Docstrings, описывающие *почему* выбрано решение.
- Используй говорящие имена переменных.
- DRY, KISS, SOLID.

# Порядок действий при запросе
1. Проверить `current_state.md` и `modding_roadmap.md`.
2. Следовать архитектуре в `architecture.md` (Data-Driven, Registry-based).
3. При добавлении контента: JSON в ресурсы + поле в класс-холдер.
4. Обновить память через `update_memory`.
