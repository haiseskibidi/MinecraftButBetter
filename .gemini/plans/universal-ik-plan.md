# План разработки: Universal IK System (Инверсная Кинематика)

## 1. Цель
Создать абстрактную, производительную и расширяемую систему инверсной кинематики, которая работает с произвольными цепочками узлов (`ModelNode`). Система должна быть отделена от конкретных имен костей (никакого хардкода "shoulder" или "hand") и поддерживать наложение ограничений (constraints).

## 2. Алгоритм: FABRIK
Для решения задачи выбран алгоритм **FABRIK** (Forward And Backward Reaching Inverse Kinematics).
- **Почему FABRIK?**
    - Поддерживает цепочки любой длины (от 2 до N костей).
    - Вычисляется итеративно: очень быстрый (подходит для Real-time).
    - Легко интегрируется с ограничениями углов (Joint Constraints).
    - Не страдает от "сингулярностей" (проблем с углами Эйлера), так как оперирует позициями векторов.

## 3. Архитектура системы

### 3.1. `IKChain` (Контейнер данных)
Описывает структуру, которую нужно согнуть.
- `List<ModelNode> nodes`: Список узлов от корня к кончику (например: Плечо -> Предплечье -> Кисть).
- `Vector3f targetPosition`: Мировая координата цели.
- `Vector3f poleTarget` (Optional): Точка "локтевого ориентира" для управления плоскостью изгиба.
- `float tolerance`: Порог точности (когда считать цель достигнутой).
- `int maxIterations`: Лимит итераций для предотвращения зависаний (обычно 10-15 достаточно).

### 3.2. `FABRIKSolver` (Логическое ядро)
Класс, реализующий математику.
1. **Forward Pass:** Тянем кости от кончика (`effector`) к цели.
2. **Backward Pass:** Тянем кости от корня (`root`) к их исходному положению.
3. **Constraint Phase:** На каждой итерации применяем ограничения углов.
4. **Baking Step:** После нахождения позиций, вычисляем необходимые `animRotation` для каждой кости, чтобы они смотрели в нужные стороны в локальном пространстве.

### 3.3. `IKConstraint` (Интерфейс ограничений)
Позволяет задавать "человеческие" пределы движения.
- `HingeConstraint`: Ограничение вращения по одной оси (например, локоть гнется только по X от -150° до 0°).
- `BallConstraint`: Ограничение вращения в конусе (например, плечо).

## 4. Интеграция

### 4.1. В Редакторе Анимаций
- **IK Mode:** При выборе кости система пытается автоматически построить цепочку вверх до ближайшего разветвления или заданного лимита.
- **Gizmo Interaction:** Манипулятор перемещает не саму кость, а `targetPosition` цепочки.
- **Live Preview:** Каждое движение мыши вызывает `Solver`, мгновенно обновляя позу модели.
- **Bake to Keyframes:** Возможность "запечь" позу IK в обычные ключи вращения (FK), чтобы сохранить совместимость с текущим движком анимаций.

### 4.2. В Игровом процессе (Procedural Animation)
- **Wall Interaction:** Если игрок стоит вплотную к стене, рука автоматически "упирается" в неё через IK.
- **Ground Alignment:** Ноги существ подстраиваются под наклон поверхности (ступни стоят на земле, а не висят в воздухе).

## 5. Формат данных (JSON)
Поддержка описания IK-цепочек в файлах моделей:
```json
"ik_chains": [
  {
    "id": "right_arm",
    "nodes": ["shoulder_r", "forearm_r", "hand_r"],
    "constraints": [
       { "node": "forearm_r", "type": "hinge", "axis": [1,0,0], "min": -150, "max": 0 }
    ]
  }
]
```

## 6. Этапы реализации

### Этап 1: Базовая математика (MVP)
- [ ] Создать класс `FABRIKSolver` в `com.za.zenith.engine.graphics.model.ik`.
- [ ] Реализовать алгоритм для свободных цепочек (без ограничений).
- [ ] Написать юнит-тест, проверяющий, что кончик цепи достигает цели.

### Этап 2: Ограничения (Joint Constraints)
- [ ] Реализовать `HingeConstraint` (самое критичное для локтей и коленей).
- [ ] Добавить в `Solver` фазу коррекции позиций согласно ограничениям.

### Этап 3: Редактор и Визуализация
- [ ] Добавить `IKManager` в `EditorInputHandler`.
- [ ] Реализовать визуальный таргет (кубик или сфера) для управления IK.
- [ ] Кнопка "Bake IK" для создания стандартных Keyframes.

### Этап 4: Полировка
- [ ] Добавить поддержку `Pole Target` (колени/локти не должны смотреть внутрь тела).
- [ ] Интеграция с JSON-загрузчиком моделей.
