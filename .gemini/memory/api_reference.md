# Developer API Guide: Zenith
Этот файл — **Single Source of Truth** (SSOT) по практическому применению внутреннего API движка Zenith. 
**ОБЯЗАТЕЛЬНО** сверяйся с ним при добавлении нового контента (Блоки, Предметы, UI, Журнал).

---

## 1. Core API & Registries

### Идентификаторы и Динамические ID (Identifiers & Dynamic IDs)
Любой контент в игре (блоки, предметы, анимации, рецепты, UI) идентифицируется через класс `Identifier`.
Движок Zenith использует систему **динамических числовых ID**. Это значит:
- В JSON-файлах поле `"id"` является **необязательным** (и крайне не рекомендуется для нового контента).
- Числовые ID назначаются автоматически при загрузке на основе порядка файлов в `.index` и их `Identifier`.
- Это исключает коллизии (конфликты номеров) и упрощает добавление новых предметов.

Никогда не используй магические числа или сырые строки там, где ожидается ID.
```java
// Правильно
Identifier myId = Identifier.of("zenith:grass_block");
int numericalId = BlockRegistry.getId(myId); // Получение динамического ID в рантайме
```

### Авто-регистрация (Classes-holders)
Для быстрого доступа к контенту в Java не нужно вручную искать его в реестре. Классы `Blocks.java`, `Items.java`, `PrefabManager.java` заполняются автоматически через рефлексию (имена полей мапятся на JSON).
```java
// Blocks.java
public static Block grass_block; // Автоматически заполнится блоком "zenith:grass_block"
```

## 2. Data-Driven Контент (JSON)

Весь контент описывается в файлах внутри `src/main/resources/zenith/`. Хардкод параметров в Java **ЗАПРЕЩЕН**.

### 2.1. Блоки (`blocks/`)
Блоки поддерживают визуальный масштаб, тинт, `VoxelShape` для коллизий, а также сложную логику добычи (Weak Spots) и прогрессивного выпадения лута (Dismantling/Scavenging).

**Базовый пример (`grass_block.json`):**
```json
{
    "identifier": "zenith:grass_block",
    "translationKey": "block.zenith.grass_block",
    "solid": true,
    "transparent": false,
    "hardness": 0.6,
    "interaction_cooldown": 0.1, // Ограничение на скорость добычи
    "textures": {
        "top": "grass_block_top.png",
        "bottom": "dirt.png",
        "side": "grass_block_side.png"
    },
    "requiredTool": "shovel",
    "tags": ["zenith:tinted"] // Включает биомный цвет (листва/трава)
}
```

**Продвинутая механика: Weak Spots и Progressive Drops (`rusty_car_chassis.json`):**
Для того чтобы блок разрушался через "слабые точки", а лут выпадал при *каждом успешном ударе* (а не только при разрушении всего блока), используется `mining_logic` и флаг `drop_on_hit` в `drops`:
```json
{
  "identifier": "zenith:rusty_car_chassis",
  "hardness": 6.0,
  "requiredTool": "crowbar",
  "breaking_pattern": "stone", // Процедурный паттерн разрушения (вместо текстуры трещин)
  "mining_logic": {
    "strategy": "weak_spots", // Включает генерацию слабых точек (миниигра при добыче)
    "precision": 0.18,        // Насколько точно нужно попасть курсором
    "miss_multiplier": 0.2,   // Штраф к урону при промахе (20% от базового урона)
    "weak_spot_color": [0.8, 0.4, 0.1] // Цвет слабых точек
  },
  "drops": [
    {
      "item": "zenith:scrap_metal",
      "chance": 0.25,         // Шанс выпадения (25% за удар)
      "tool": "crowbar",      // Требуемый инструмент для выпадения этого лута
      "drop_on_hit": true     // КЛЮЧЕВОЙ ФЛАГ: лут вылетает в момент удара по блоку
    }
  ]
}
```
### 2.2. Предметы (`items/` и Компоненты)
Предметы **НЕ наследуются** (никаких `ToolItem` или `FoodItem`). Все предметы — это базовый класс `Item`, а их поведение определяется компонентами.

**Базовый пример (`stone_axe.json`):**
```json
{
  "identifier": "zenith:stone_axe",
  "texture": "zenith/textures/item/stone_axe.png",
  "visualScale": 1.2, // Увеличивает размер 3D-модели в руке
  "weight": 1.5,      // Влияет на инерцию и пружинную физику
  "components": {
    "zenith:tool": {
      "type": "axe",
      "efficiency": 1.5,
      "durability": 120
    },
    "zenith:thermal": { // Позволяет предмету раскаляться
      "initialTemperature": 20.0,
      "burnThreshold": 60.0
    }
  }
}
```

**Другие важные компоненты:**
- **Рюкзаки и экипировка:**
```json
"components": {
  "zenith:bag": { "slots": 18 }, // Делает предмет контейнером (ItemInventory)
  "zenith:equipment": { "slot": "accessory", "strict": true } // Разрешает надевать в спец. слоты GUI
}
```
- **Еда (`type: food`):** В базовых полях указывается `"nutrition": 2.0` и `"saturation": 1.0`.
- **Топливо:** `"zenith:fuel": { "fuelAmount": 100.0 }`.

## 3. Инвентари и GUI v4 (`gui/`)

Система интерфейсов Zenith использует CSS-подобную JSON-верстку с относительным позиционированием.
Окно состоит из групп слотов (Group).

### Ключевые концепции
- **Юниты `"1s"`**: Означает `размер слота + spacing`. Идеально для отступов, кратных сетке инвентаря.
- **Относительное позиционирование**: `relativeTo` (какую группу взять за якорь) и `relativeAlign / relativeAlignY` (как пристыковаться).
- **Быстрое перемещение**: `quickMoveTo` определяет массив групп, куда улетят предметы при Shift+Click.
- **Strict-слоты**: Предметы с `zenith:equipment` могут требовать `strict: true` слот (например, рюкзак).

**Пример (`chest.json`):**
```json
{
  "identifier": "zenith:chest",
  "groups": [
    {
      "id": "container",
      "type": "grid",
      "inventorySource": "container",
      "cols": 9,
      "spacing": 4,
      "anchor": "center",
      "quickMoveTo": ["hotbar", "pockets"]
    },
    {
      "id": "hotbar",
      "type": "grid",
      "inventorySource": "player",
      "startIndex": 0,
      "slotsCount": 9,
      "cols": 9,
      "spacing": 4,
      "relativeTo": "container", // Привязка к сундуку
      "relativeAlign": "bottom",
      "relativeAlignX": "left",
      "fixedOffsetY": "1s", // Отступ вниз на размер 1 слота
      "quickMoveTo": ["container"]
    }
  ],
  "background": {
    "type": "solid",
    "color": [0.1, 0.1, 0.1, 0.8],
    "padding": 12,
    "includeGroups": ["container", "hotbar"] // Адаптивный фон вокруг этих групп
  }
}
```

### 4.2. Foliage Animation (v1.0)
Механика процедурного покачивания растительности на ветру.

- **JSON Property**: `"sway": true` включает анимацию для блока. По умолчанию `false`.
- **Implementation (Vertex Weights)**:
  - Движок автоматически назначает веса вершинам: `0.0` (основание) и `1.0` (верх).
  - **Additive Weights**: Для многоблочных растений (`DOUBLE_PLANT`) верхний сегмент получает веса `1.0` -> `2.0`, что гарантирует бесшовный изгиб всего стебля.
- **Selection Sync**: Хитбокс автоматически синхронизирует свою анимацию с блоком, если у того включен `sway`.
- **Uniform Control**: Юниформа `uSwayOverride` позволяет Renderer'у управлять интенсивностью анимации (например, принудительно остановить хитбокс на статичных блоках).

### 4.3. Разрушение блоков и VFX

- **MiningController** изолированно управляет кулдаунами, прогрессом и слабостями.
- **Индикация прогресса (Mining Heat VFX)**: При ударе накапливается **Heat (Жар)** пропорционально прогрессу разрушения блока. Это заставляет руки или инструмент в руках (если у него есть компонент `zenith:thermal`) визуально раскаляться и светиться красным/оранжевым цветом, тем самым показывая степень разрушения блока без UI-полоски. `MiningVFXManager` управляет этим состоянием и плавно остужает предмет при бездействии.
- **Persistent Scars**: Шрамы (зарубки) остаются на блоке навсегда, сохраняясь в `Vector4f` (где `W` - интенсивность) и плавно исчезают при регенерации блока (спустя 5 сек после удара).

### 4.2. Treecapitator (Рубка деревьев)
Любое дерево в игре разрушается постепенно.
- Для этого блок должен иметь тег `treecapitator` в JSON.
- Алгоритм отличает постройки игрока от живых деревьев по флагу **BIT_NATURAL (0x80)** в метаданных блока.
- При поиске метаданных маскируй направление: `metadata & 0x07`.

### 4.3. Actions (Действия)
Все затраты выносливости и шум вынесены в `zenith/actions/*.json`.
**Пример (`walk.json`):**
```json
{
  "staminaCostPerSecond": 0.5,
  "hungerCostPerSecond": 0.05,
  "noiseLevel": 0.3
}
```

### 4.4. Рецепты и In-World Crafting (Крафт в мире)
Для реализации крафта на пне используется интерфейс `ICraftingSurface` (например, `StumpBlockEntity`). 
Механика работает в мире (без GUI): ПКМ кладет предмет, Shift+ПКМ забирает, ЛКМ подходящим инструментом выполняет процесс.

**Пример 1: Napping (Обтёсывание по паттерну)**
```json
{
  "identifier": "zenith:unfired_vessel",
  "type": "napping",
  "input": "zenith:clay_ball",
  "result": { "item": "zenith:unfired_vessel", "count": 1 },
  "pattern": [
    " ### ",
    "#   #",
    "#   #",
    "#   #",
    " ### "
  ]
}
```

**Пример 2: Carving (Вырубание)**
```json
{
  "identifier": "zenith:oak_log_carving",
  "type": "carving",
  "input": "zenith:oak_log",
  "tool": "zenith:stone_knife",
  "intermediate": "zenith:unfinished_stump",
  "result": "zenith:stump"
}
```

### 4.5. Установка блоков и Foraging
- **Превью установки:** Голограмма (Preview) устанавливаемого блока отображается **только когда игрок крадется** (Sneaking). Это предотвращает визуальный шум при паркуре.
- **3D Foraging:** Мелкие ресурсы на земле (палки, камни) реализованы не как блоки, а как физические сущности (`ResourceEntity`). Они собираются мгновенным кликом (ПКМ/ЛКМ) без кулдауна разрушения.

## 5. Survivor's Tablet (Дневник)

Добавление новой статьи требует строгой последовательности действий.

### Алгоритм:
1. Создать JSON в `journal/entries/` (например, `pottery.json`).
2. Добавить имя файла (`pottery.json`) в `journal/entries/.index`.
3. Добавить Identifier (`zenith:pottery`) в массив `entries` соответствующей категории (например, `survival.json`) в `journal/categories/`.
4. Проверить переводы ключей в `lang/ru_ru.json`.

### Поддерживаемые элементы (`elements`):
- `text`: Обычный текст (ключ локализации).
- `header`: Заголовок.
- `tip`: Блок-подсказка с иконкой.
- `image`: Картинка.
- `item_row`: Отображение нескольких предметов в ряд `{"type": "item_row", "items": ["zenith:flint", "zenith:stone"]}`.
- `recipe`: Отображение сетки крафта `{"type": "recipe", "value": "zenith:campfire"}`.

**Пример статьи:**
```json
{
  "identifier": "zenith:pottery",
  "title": "journal.entry.pottery.title",
  "icon": "zenith:fired_vessel",
  "elements": [
    { "type": "header", "value": "journal.entry.pottery.step1" },
    { "type": "recipe", "value": "zenith:unfired_vessel" },
    { "type": "tip", "value": "journal.entry.pottery.tip" }
  ]
}
```

## 6. Физика, Анимации и Рендеринг

### 6.1. Рендеринг от первого лица (Viewmodel System)
- **Иерархия костей:** Рендеринг рук и предметов осуществляется через скелетную систему (`Shoulder -> Forearm -> Hand`). 
- **PCA Авто-ориентация:** `ItemMeshGenerator` использует метод главных компонент (PCA) для анализа пикселей 2D-текстуры и автоматического вертикального разворота предмета.
- **Вес (Weight) и Инерция:** Поле `weight` в JSON предмета определяет силу пружинной физики (чем тяжелее, тем больше предмет отстает от камеры при поворотах).
- **Procedural Hand Conditions:** Руки (Viewmodel) могут динамически покрываться процедурной кровью, грязью и влагой (`hand_conditions.glsl`), которые накладываются пиксельно-точно поверх 3D-модели (Splatters) в зависимости от действий игрока.

### 6.2. Рендеринг мира (World Rendering)
- **Texture Array:** Текстуры загружаются в массив `GL_TEXTURE_2D_ARRAY`, решая проблему мерцания швов на стыках (edge bleeding).
- **VoxelShape Hightlights:** Контур выделения блока строится динамически из массива AABB блоков, автоматически сшивая внутренние грани для идеального отображения (например, ступеней).

### 6.3. Интерполяция и Сущности (Entity System)
- **144Hz+ Физика:** Все сущности (игроки, мобы, предметы) хранят `prevPosition` и `prevRotation`. Отрисовка происходит с интерполяцией (через коэффициент `alpha`), сглаживая фиксированный физический тик.
- **Ground Lock:** Выпавшие предметы (ItemEntity) отключают влияние гравитации, находясь на земле, чтобы предотвратить дрожание модели внутри блоков.

## 7. Индустриальная система (Tier 1)

Блоки, оперирующие энергией (генераторы, батареи, провода, лампы), взаимодействуют через интерфейс `IEnergyStorage`. Передача энергии вынесена в `BlockEntity` и происходит каждый физический тик (`ITickable`). Поиск соседей осуществляется только через цикл по `Direction.values()`.

## 8. Прочие Data-Driven Механики

### 8.1. Паркур и Локомоция
Анимации движения вычисляются через Data-Driven кривые. 
- Система автоматически тратит выносливость (Stamina) из `actions.json` при состояниях `HANGING` и `CLIMBING`.
- Прицеливание на уступ (`Ledge Detection`) использует рейкаст, а подъем происходит по кривой Безье (Bezier 3D Path).

### 8.2. Двойные растения (Double Plants)
Для блоков высотой в 2 блока (например, высокая трава) используется:
- `PlacementType.DOUBLE_PLANT` в JSON блока.
- В метаданных: `0` для нижней части, `1` для верхней.
- Поле `upperTexture` в JSON автоматически применяется рендерером для верхней половины.

### 8.3. Динамические прицелы (Crosshairs)
Отрисовываются через матрицы 1/0 в JSON. Поддерживают анимации, которые настраиваются прямо в конфигурации:
- `recoilScale`: отдача (увеличение) при ударе.
- `spreadScale`: разлет элементов при прогрессе добычи блока.

### 8.4. Декоративные Сущности (Entity Registry)
Вместо создания тяжелых блоков для мелких декоративных объектов (например, лежащие бревна, палки), Zenith использует легковесные сущности из реестра `EntityRegistry`.
Конфигурация (`zenith/entities/log_pile.json`):
```json
{
    "identifier": "zenith:log_pile",
    "modelType": "block",
    "texture": "zenith:oak_log",
    "visualScale": [0.55, 0.55, 0.95],
    "hitbox": [0.55, 0.55, 0.95]
}
```

### 8.5. Data-Driven Структуры (Structures)
Генерация построек и зданий описана в `src/main/resources/zenith/structures/`.
Используется матричная система по слоям (`layers`) с привязкой символов к блокам через `palette`.
Пример палитры:
```json
"palette": {
  "C": "zenith:cyan_concrete",
  ".": "zenith:air",
  " ": "-1" // Пропустить (не заменять существующий блок)
}
```

### 8.6. Условные группы в GUI (Conditions)
В `gui/*.json` группы слотов могут появляться динамически в зависимости от стейта (например, открыта сумка или включен dev-режим) с помощью поля `"condition"`.
Пример из `player_inventory.json`:
```json
"condition": "has_pouch" // Группа отрендерится только если есть предмет-сумка
```

## 9. Глобальные Настройки и Реестры (Registry)

Для тонкой настройки движка без перекомпиляции используются глобальные конфигурации в папке `src/main/resources/zenith/registry/`:

### 9.1. Физика Игрока (`physics.json`)
Хранит все ключевые константы локомоции: хитбоксы (`standingHeight`, `sneakingHeight`), дистанцию взаимодействия (`grabDistance`), высоту паркура (`maxGrabHeight`), а также множители скорости и базовый кулдаун добычи (`baseMiningCooldown`). Изменение этого файла меняет физику налету.

### 9.2. Спавн ресурсов на земле (`scavenge.json`)
Конфигурирует вероятность появления ресурсов (сущностей из `EntityRegistry`) на поверхности блоков при генерации мира:
```json
[
  { "block": "zenith:stick", "chance": 0.015 },
  { "block": "zenith:rock", "chance": 0.01 }
]
```

### 9.3. Типы древесины (`wood_types.json`)
Массив идентификаторов базовых бревен. Используется в системе Treecapitator (см. п. 4.2) для маппинга индекса из метаданных (0-5) в конкретный блок при выпадении лута, позволяя всем стадиям рубки использовать одни и те же технические блоки.

## 10. Animation Studio (Редактор анимаций)

В движок встроен полноценный редактор анимаций (Toggle по **F8**), работающий в изолированной 3D-сцене.
- **Архитектура:** Построена на чистой прямой кинематике (FK) с использованием 3D-Gizmos (Стрелки/Кольца). 
- **Auto-Keying:** Создание/обновление ключевых кадров происходит автоматически при подтверждении трансформации или клике.
- **FABRIK IK:** Поддержка инверсной кинематики (IK) с Pole Targets для локтей и коленей. Цепи настраиваются автоматически (`autoSetup`), а затем запекаются (Baking) в стандартные Euler-ключи для экспорта.
- **Интеграция:** Созданные файлы сохраняются как JSON и загружаются через `AnimationRegistry`.

## 5. RPG System & Items
Система предметов построена на компонентах (ItemComponents) и динамических реестрах характеристик.

### Реестры (Registries)
- `StatRegistry`: Регистрация всех игровых параметров.
- `RarityRegistry`: Определение уровней редкости (цвета, количество слотов аффиксов).
- `AffixRegistry`: База данных префиксов и суффиксов.

### Item JSON Structure
Пример расширенного предмета:
```json
{
  "identifier": "zenith:scrap_axe",
  "rarity": "zenith:uncommon",
  "gender": "masculine",
  "tags": ["zenith:tools", "zenith:melee"],
  "components": {
    "lootbox": {
      "loot_table": "zenith:equipment",
      "opening_time": 1.0,
      "rarity_weights": { "zenith:common": 40, "zenith:uncommon": 60 }
    }
  }
}
```

### Принципы работы
1. **Гендерная локализация**: Используйте суффиксы `.m`, `.f`, `.n` в `ru_ru.json` для автоматического склонения аффиксов.
2. **Data-Driven Tooltips**: Тултипы собираются автоматически на основе имеющихся в `ItemStack` компонентов и статов. Максимальная ширина — 280px.
3. **Loot Tables**: Находятся в `resources/zenith/registry/loot_tables/`. Поддерживают вложенные пулы и веса.
4. **Stats Calculation**: Итоговый стат предмета = `Base (из JSON)` + `Modifiers (от аффиксов)`.
5. Formatting: Для текста в коде и JSON используются Minecraft-style коды через символ `$`.

## 6. HUD Configuration (`gui/hud.json`)
Все элементы игрового оверлея управляются через `hud.json`.

### Параметры элементов
- `type`: `text` или `bar`.
- `anchor`: Позиционирование относительно экрана (`bottom`, `top`, `bottom_center` и т.д.).
- `x`, `y`: Смещения в `px` или `%`.
- `fontSize`, `minFontSize`, `maxWidth`: Параметры для динамического масштабирования текста.

### Логика масштабирования
Если ширина текста (например, названия предмета) превышает `maxWidth`, движок будет автоматически уменьшать шрифт до тех пор, пока он не влезет в границы, либо пока не достигнет `minFontSize`.
